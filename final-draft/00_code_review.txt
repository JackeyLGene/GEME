二、代码与声称的匹配度分析
======================================================================

2.1 "概念形成"机制（论文声称：系统通过竞争记忆形成概念）
----------------------------------------------------------------------
实际代码行为：
  observe() 接受向量和签名，在已有帧中找最近邻。
  若距离小于自适应阈值 → 合并（加权质心，权重+1）。
  否则 → 创建新帧，容量满时淘汰最低得分帧（weight - age×0.1）。

评估：机制存在且有效。自适应阈值（引导期Q1，之后中位数）
比固定阈值更鲁棒。代码行 99-115 确认。

问题：
  - 【中等】淘汰评分使用 age×0.1（行 149），但 induction_clean
    使用 age×0.05（行 235）。两个不同的年龄衰减值，无注释说明原因。
  - 【微小】淘汰排序属性不一致：行 149 用 frame.weight - frame.age*0.1，
    行 235 用 frame.weight - frame.age*0.05。两个不同的衰减率。

声称评估：验证通过，但年龄衰减系数不一致需澄清。


2.2 "壁检测"（论文声称：系统在签名空间边界碰到不可区分性）
----------------------------------------------------------------------
实际代码行为：
  evaluate_sig() 计算待测签名与帧签名的重叠率。
  overlap = |parts ∩ frame_parts| / min(|parts|, |frame_parts|)
  若 ≥ 0.75 且帧权重大于中位数 → 返回 S2（已涌现）。

  对实验二（所有输入 eq_succ）：
    有序和随机输入产生的签名完全相同 → overlap 完全相同
    → evaluate_sig 必然返回相同结果
    → 壁是结构性的，非数据/阈值伪影。

问题：
  - 【无】代码行为与论文声称完全一致。壁确实是 evaluate_sig()
    签名重叠检查的直接结果。这是结构性而非实现缺陷。

声称评估：验证通过。壁是签名匹配逻辑的直接推论。


2.3 "共现自指"（论文声称：滑动窗口追踪共现模式，形成关联帧）
----------------------------------------------------------------------
实际代码行为：
  窗口内每对相距 ≤3 步的条目，其签名对被记录（行 157-163）。
  当 cooccur[key] / total_steps ≥ thresh（默认 0.15）时，
  创建关联帧（签名 = sa+"──"+sb，行 171）。

问题：
  - 【重大】步数计数器（行 154）使用 len(self._window) 作为步数。
    但窗口长度固定为 _win_max（默认 80）。当窗口满时，步数计数器
    不递增——它回滚。len(self._window) 永远 ≤ 80。这意味着
    在稳态下，step_id 是循环的，最多 80 个唯一值。
    
    更关键的是：行 161 的 `if id1==id2: continue` 本意是跳过
    同一步内的自配对。但 step_id 被限制在 1-80 循环。
    当窗口满时，每 80 步后 step_id 重置为 1。
    两对不同步数但同 step_id 的条目会被错误跳过。

    实际影响评估：实验中的输入量（600-1200）远大于窗口（80），
    所以 step_id 会循环多次。这可能导致某些应该被计数的共现
    对被错误跳过，但不会漏掉所有——因为同一签名对不仅出现一次。
    这是一个错误，但可能不影响统计结果的可重复性。

  - 【中等】关联帧创建时（行 180-195），若 base_vecs 不足 2
    （即没有两层下层帧同时匹配 sa 和 sb），向量被置为零向量
    （行 195）。但权重仍然设定为 count（正数）。这会产生一个
    有权重但无意义的帧。后续合并到此帧的向量会与零平均，降低
    合并质量。

声称评估：部分验证通过。存在步数计数器循环问题和零向量问题。


2.4 "词汇自生"（论文声称：系统自动从第一层签名提取内容词）
----------------------------------------------------------------------
实际代码行为：
  代码中不存在词汇生成的引擎级机制。词汇生成在实验脚本中，
  不是 GEME 引擎的一部分。

  脚本流程（以 _exp_loop_close.py 为例）：
    1. 跑 GEME1
    2. 查询 GEME1.memory.frames 找到最高权重帧
    3. 解析帧签名为字符串部件，提取内容词（剔除 eq/and 等通用词）
    4. 扫描窗口检测爆裂大小
    5. 将 content_word + "_" + str(burst_size) 作为函数名传入 GEME2

问题：
  - 【重大】论文声称"系统自动提取"——但不是引擎自动做的。
    是实验脚本在 GEME1 跑完后，读取了 memory 的内部状态，
    然后手动构造了新函数名，再传给新的 GEME 实例。
    这是"脚本级桥接"（脚本是独立的 Python 文件，非引擎内嵌），
    不是"系统自生"。

  - 论文在 Methods 4.3 中诚实陈述了这一点，但 ABSTRACT 的
    "generates its own vocabulary from the first layer's internal
    statistics" 具有误导性——这说的是引擎行为，但实际上是
    脚本行为。

声称评估：ABSTRACT 声明过度。METHODS 诚实。需要修正 ABSTRACT。


2.5 "链帧定理"（论文声称：链帧作为原型定理生成器）
----------------------------------------------------------------------
实际代码行为：
  链帧在 observe() 行 202-221 创建。条件：
    - 已存在的关联帧权重 ≥ 20
    - chain_count < max_chains
    - 另一帧有关联签名（──）且非链签名（══）
    - 两个关联共享元素且至少一个关联有 ≥ 2 个独特元素
    → 创建签名 = assoc_sig+"══"+other_sig

问题：
  - 【中等】链帧仅在 observe() 中被现有关联触发时创建。
    没有主动的链帧推理——它是对共现事件的被动响应。
    论文用"定理"和"推理"来描述链帧，但代码不做推理。
    它只是把两个共享元素的关联签名用 ══ 拼接。
    更像"共现的共现"而不是"推理"。

  - 【微小】行 203 的 existing[0].weight > 20 硬阈值。
    如果关联帧权重从未超过 20（小数据集时可能），链条从不形成。

声称评估：链帧被描述为"定理生成器"过度。它确实是帧关系，
但不是主动推理。建议改为"concept-relation encoding"。

三、代码与论文数据一致性验证
----------------------------------------------------------------------

3.1 随机种子的可复现性
  _random.Random(42) 在实验脚本顶部使用。Python 的 random 模块
  在种子固定时是确定性的。实验声称的 50/100 轮应可完全复现。

3.2 声称的权重数值
  论文声称 assoc_w mean=290.4 sd=0.0（实验二，50轮）。
  由于种子 42-91 是线性序列，random.Random 对同一序列的行为
  在不同种子下可能产生极大差异。sd=0.0 表明 50 轮结果完全相同。
  这暗示要么种子范围内的随机性不足以引起权重差异，要么系统行为
  被某个主导因素锁定（如签名全部相同）。

  代码分析：实验二所有输入确实共享 eq_succ 签名，所以帧签名是
  一样的，合并行为被签名锁定。sd=0.0 不是数据造假——是签名一致
  性的可预测结果。建议在论文中解释这一高稳定性。

3.3 实验三的 L2 方差（sd=222.4 vs mean=237.4）
  高方差与论文声称一致：种子不同导致 burst 分布不同。代码确认
  词汇生成部分依赖于窗口扫描，而窗口内容受随机种子影响。

2.7 VocabularyModule 审查（新增模块，50行）
----------------------------------------------------------------------
位置：src/gira/phase6/vocabulary_module.py（独立文件，非GEME引擎的一部分）

代码结构：
  一个类（VocabularyModule），三个公共方法（analyze、get_formulas、
  has_generated、vocabulary属性），一个私有方法（_extract_vocabulary）。

审查结果：

  - 【无】analyze() 方法检测壁条件的逻辑：高权重帧签名是否收敛为
    单一签名。与论文声称一致——壁检测有效（100轮验证100%）。
    
  - 【无】_extract_vocabulary() 的内容词提取：从最高权重帧签名中
    解析第一个非通用词。简单、透明、有效。对 "eq_conn──eq_conn"
    提取 "conn"——正确。

  - 【中等】窗口爆裂检测（行83-95）：扫描_window中的所有条目。
    注意：此扫描是对已有数据的后处理，不是流式处理。因此
    analyze() 应该在一个GEME完成运行后调用，不在运行中调用。
    这不是错误——它与"词汇模块在感知层完成工作后介入"的设计
    一致。方法与设计一致，但论文需明确说明使用时机。

  - 【中等】get_formulas() 在方法内部import（行114-115）。
    这是延迟导入（lazy import），通常用于避免循环依赖。
    本模块没有任何循环依赖——建议移到文件顶部，提高可读性。

  - 【微小】analyze() 的形参名为 memory，但实际参数通常是
    GEME实例的.memory属性。如果用户传入GEME实例本身会报错
    （GEME没有.frames属性）。建议增加类型提示或文档说明。

对比旧方法（脚本级桥接）：
  旧：实验脚本读取GEME1.memory.frames → 手动构造字符串 →
      硬编码到GEME2的输入。
  新：VocabularyModule.analyze(GEME1.memory) → 自动返回词汇表 →
      get_formulas(数据) → 生成GEME3的输入。

新模块将之前的"手动步骤"自动化，但词汇生成的逻辑本质未变。
它不是"新的认知理论"——是"更干净的工程实现"。这一点论文需要
诚实说明。

模块代码评级：B+（简洁、功能正确、文档充分）
